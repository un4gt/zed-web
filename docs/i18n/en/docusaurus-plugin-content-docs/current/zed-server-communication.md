This document uses zed-server as shorthand for the `zed-remote-server proxy` process on the target machine. The browser never connects to zed-server directly. Every request goes through the gateway, and the gateway manages remote processes and file operations over SSH.

## Session Creation

The frontend calls the gateway session API with the SSH target, port, user, project path, and remote server policy. The gateway then:

1. Validates `host` and `project_path`.
2. Selects a remote server version using `latest`, `pinned`, or `disabled`.
3. Probes the target machine over SSH and starts `zed-remote-server proxy`.
4. Initializes the Zed proxy client and adds the project path as a worktree.
5. Returns a session snapshot. The frontend then opens the command WebSocket with that session id.

## Command WebSocket

The editor and project tree share one command channel:

```text
/api/sessions/{session_id}/commands
```

Each request is a JSON envelope:

```json
{
  "id": "1",
  "type": "buffer.open",
  "payload": {
    "path": "src/main.rs"
  }
}
```

The gateway returns one or more responses with the same `id`. Terminal responses usually have a `type` ending in `.complete`; failures use `type: "error"`.

## Buffer Open

`buffer.open` opens an editable text buffer. The gateway prefers the zed-server proxy and falls back to SSH file reads on failure or timeout.

Response order:

```text
buffer.open.started
buffer.chunk*
buffer.open.complete
```

`buffer.chunk` transfers base64-encoded text chunks. `buffer.open.complete` returns:

- `bytes_read`
- `truncated`
- `read_only`
- `resource_version`

`resource_version` is opaque to the frontend. Current schemes are:

- `zed-vector-clock`: Zed buffer version from zed-server.
- `ssh-stat`: content version generated by the SSH fallback path.

## Buffer Save

When saving, the frontend sends `buffer.save` with incremental changes only, not the full file:

```json
{
  "path": "src/main.rs",
  "base_resource_version": {
    "scheme": "zed-vector-clock",
    "value": "opaque"
  },
  "batches": [],
  "expected_content_length": 1280
}
```

The gateway reads the current remote version and compares it with `base_resource_version`:

- If versions match, it applies text edits in batch order and saves.
- If versions differ, it returns `status: "conflict"` and does not write the remote file.

A successful save returns `applied_seq`, `bytes_written`, and the new `resource_version`. A conflict response includes the current remote version when available and leaves the local dirty buffer intact.

## Buffer Sync

`buffer.sync` lets the frontend check remote state after reconnect, restore, or explicit sync. Each request item includes path, base version, dirty state, and last seq. Response statuses are:

- `unchanged`: remote version did not change.
- `remote_changed`: remote version changed.
- `missing`: the gateway could not read the path.

## Related Commands

- `tree.list`: lists the project tree.
- `session.reconnect`: restarts the remote proxy with the same session identifier.
- `file.open` / `file.save`: legacy compatibility fallback. New editor flows use `buffer.*`.

## Design Constraints

- AI edits, user input, and restored dirty buffers must enter the working copy before saving through `buffer.save`.
- The frontend does not bypass the gateway to write files.
- When remote content changes, the gateway does not auto-merge and does not overwrite the remote file.
